# 音声フリーズ修正計画

「話す」「聞く」で発生するフリーズ問題の段階的修正計画。原因分析の詳細はこの文書末尾の「根本原因」節に記す。

---

## 問題の再掲

| 症状 | 発生頻度 |
|-----|---------|
| 「話す」単独で固まる | 高 |
| 「聞く」+「話す」同時で固まる | 非常に高 |

---

## フェーズ 1（本フェーズ）— 「話す」単独フリーズの解消

### スコープ
サーバ側 [`AudioPlayer`](../server/audio.py) を専用ワーカースレッド + 有界キューで非同期化し、`OutputStream.write()` を Socket.IO のイベントハンドラスレッドから完全に切り離す。

### 変更内容
- `AudioPlayer.play()` を **非ブロッキング**化（PCM を内部キューに投入するだけ）。
- 新設の writer スレッドがキューを取り出して `stream.write()` を実行。
- 有界キュー（容量 16 チャンク ≒ 約 1 秒）、満杯時は古いチャンクを捨てる（雪だるま防止）。
- PortAudio エラー時は writer スレッド内で `OutputStream` を再オープン。

### 期待効果
- `audio_talk` ハンドラがデバイス I/O で詰まらない → WS 接続全体が固まらない。
- 遅延が積算しても上限で頭打ち（古いチャンク廃棄により real-time に近い再生を維持）。

### 検証
- 「話す」を 30 秒以上連続使用してもボタン応答性が保たれること。
- サーバログに `AudioPlayer: playback error` などのスタックトレースが出ないこと。

### 影響範囲
- 変更: [server/audio.py](../server/audio.py)
- 他モジュールの API は変更なし（`play()` のシグネチャは保持）。

---

## フェーズ 2 — 「聞く」+「話す」同時フリーズの解消

### 対策候補
1. **クライアント側のローカルフィードバック遮断**
   - [static/js/audio.js](../static/js/audio.js) で `processor.connect(audioCtx.destination)` を `GainNode(0)` 経由に変更し、自分のマイク音が自端末スピーカに流れないようにする。
2. **PortAudio の入出力競合緩和**
   - 同一デバイスでの並行 I/O を見直す（`sd.Stream` の duplex 化、または明示的なホスト API 選択）。
3. **自エコー抑止**
   - 同一 IP から「聞く」+「話す」が同時に有効なときは、サーバスピーカ出力をその IP に配信しない／減衰させる方針を検討。
4. **送信側バックプレッシャ**
   - [`_stream_audio_to_client`](../server/app.py) の emit 詰まりを検知し、リスナのキューを早めにドロップさせる。

### 検証
- 聞く + 話すを 60 秒同時使用してフリーズしないこと。
- フィードバック音が増幅しないこと。

## フェーズ 3（将来）— 追加の安定性施策

## 根本原因（参考）

1. **同期ブロッキングな `OutputStream.write()` を Socket.IO ハンドラスレッドで呼んでいる**（最大要因）
   - [server/audio.py](../server/audio.py) `AudioPlayer.play()` → `self._stream.write(samples)` がブロックすると、その WS 接続の全イベントが詰まる。
   - 再生キューがないため、瞬間的なデバイス遅延が積算する。
2. **PortAudio の入出力デバイス競合**（同時利用で顕在化）
3. **クライアント側のローカルフィードバック** — `processor.connect(audioCtx.destination)` で自分の声をスピーカに出している。
4. **サーバ側に AEC がなく、聞く+話す同時利用で音響フィードバックループが発生**しデータ量が常時最大化。
